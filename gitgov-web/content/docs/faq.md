## General

### What is GitGov?

GitGov is a **Git governance platform** that captures operational metadata from developer workstations (commits, pushes, branches) and sends it to a central Control Plane for audit, compliance, and policy monitoring. It does not read, analyze, or transmit source code.

### Who is GitGov for?

GitGov is designed for **engineering teams and organizations** that need:

- Audit trails for regulatory compliance (SOC 2, ISO 27001, PCI-DSS).
- Visibility into development operations without inspecting code.
- Policy enforcement awareness (e.g., protected branch rules, working hours).
- CI/CD traceability linking commits to Jenkins pipelines and Jira tickets.

### Is GitGov open source?

GitGov is developed by **Yohandry Chirinos**, a Venezuelan software engineer with over 10 years of experience in diverse enterprise environments, banking, fintech, and startups. GitGov is not open source — it emerged as a software product designed to improve operational traceability for organizations.

## What GitGov Does NOT Do

This is one of the most important sections. Understanding what GitGov **does not** do is essential for developers and decision-makers alike.

### Does GitGov read my source code?

**No.** GitGov captures only metadata: event type, commit SHA, branch name, author, timestamp, file count, and repository name. Source code, file contents, diffs, and commit message bodies are **never transmitted** and never leave the developer workstation.

### Does GitGov monitor my screen, keystrokes, or applications?

**No.** GitGov only observes Git operations (commit, push, branch creation). It has no access to your screen, clipboard, browser, IDE, or any application outside of Git.

### Does GitGov analyze code quality or run static analysis?

**No.** GitGov does not lint, review, or assess the quality of your code. It captures metadata about *when* and *where* Git events happen, not *what* the code contains.

### Does GitGov replace CI/CD?

**No.** GitGov integrates with CI/CD tools (Jenkins, GitHub Actions) to correlate commits with pipeline results. It **does not** run builds, tests, or deployments.

### Does GitGov block Git operations?

**No.** GitGov is a **detection and observability** tool, not an enforcement tool. It can flag that a push to a protected branch occurred, but it does not prevent the push from happening.

### Does GitGov make HR or disciplinary decisions?

**No.** Signals generated by GitGov are **advisory observations** — they indicate a policy rule was triggered. Signals do not establish intent, negligence, or fault. The deploying organization is fully responsible for any decisions made based on signals.

### Does GitGov profile individual developer productivity?

**No.** There are no "lines of code per day", "commit frequency scores", or any productivity rankings. GitGov is a governance and compliance tool, not a performance tracker.

### Does GitGov sell or share my data?

**No.** All data belongs to the deploying organization. GitGov has no data monetization model. Data is not shared with third parties.

### Does GitGov store passwords or secrets?

**No.** API keys are hashed (SHA-256) before storage. GitGov never collects, stores, or transmits passwords, tokens, or `.env` values from your workstation.

## Data & Security

### Where is my data stored?

Event data is stored in a **PostgreSQL database** controlled by the deploying organization (Supabase-managed or self-hosted). The desktop app maintains a local SQLite outbox for offline resilience. See [Security Policy](/docs/security) for full details.

### Is my data encrypted?

**Yes, at multiple layers:**

- **In transit:** TLS (HTTPS) between Desktop and Control Plane.
- **At rest:** AES-256 disk encryption on Supabase-managed databases; API keys stored as SHA-256 hashes.
- **On the workstation:** API keys stored in the OS keyring (Windows DPAPI, macOS Keychain, Linux Secret Service).

### Can audit records be modified or deleted?

**No.** Audit event records are **append-only** by design. The API does not expose UPDATE or DELETE operations on event tables. Every data export is itself logged as an audit event.

### Who can see my events?

Access is controlled by **role-based access control (RBAC)**:

- **Developers** see only their own events.
- **Admins** see all events, statistics, and dashboard data.
- There is no way for a Developer-role user to access another developer's events.

### How are API keys protected?

API keys are hashed with SHA-256 before being stored in the database. The plaintext key is only shown once at creation time and is never persisted on the server. On the desktop, keys are stored in the OS credential manager (keyring).

## Desktop App

### What platforms does GitGov Desktop support?

GitGov Desktop is built with **Tauri** and supports **Windows**, **macOS**, and **Linux**.

### What happens if I lose internet connectivity?

Events are queued in a **local SQLite outbox** and automatically synced when connectivity is restored. No events are lost.

### Does GitGov require admin privileges to install?

No. GitGov Desktop installs in user space and does not require elevated permissions on most systems.

### How do I configure governance policies?

Policies are defined in a `gitgov.toml` file in your repository root. See [Governance & Policies](/docs/governance) for the full configuration reference.

## Control Plane & Server

### What is the Control Plane?

The Control Plane is the central **Axum (Rust) server** that receives events from desktop clients, processes webhooks from GitHub/Jenkins/Jira, runs policy checks, and serves the admin dashboard.

### Can I self-host the Control Plane?

**Yes.** The Control Plane can be deployed on any server that runs Rust binaries. It requires a PostgreSQL database. See [Connect to the Control Plane](/docs/control-plane) for setup instructions.

### What integrations are supported?

- **GitHub** — webhooks for push and branch events, audit log streaming.
- **Jenkins** — pipeline event ingestion and commit-to-pipeline correlation.
- **Jira** — ticket ingestion, commit-to-ticket correlation, and coverage reports.

## Compliance

### Does GitGov help with SOC 2 compliance?

**Yes.** GitGov provides append-only audit trails, role-based access control, and immutable event records — all key controls for SOC 2 Type II engagements.

### Does GitGov help with GDPR compliance?

GitGov is designed with GDPR principles:

- **Data minimization** — only metadata, never source code.
- **Right of access** — developers can view their own events.
- **Portability** — `POST /export` provides machine-readable data export.
- **Controller/processor distinction** — the deploying org is the controller; GitGov is the processor.

See [Privacy & Signal Liability](/docs/privacy) for the full GDPR reference.
